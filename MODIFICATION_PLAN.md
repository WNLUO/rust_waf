# rust_waf 改造方案

## 0. 执行状态（2026-04-11）

当前改造已完成，落地结果如下：

- [x] P0. 收缩 L7 能力边界
- [x] P0. 为管理面增加安全控制
- [x] P1. 实现运行时热更新
- [x] P1. 建立更真实的测试体系
- [x] P2. 清理配置与页面结构

本轮实际完成情况：

- 后端不再把本项目表述为本地 “L7 检测器”，`L7Inspector` 已收缩为纯 HTTP 接入处理能力，协议接入与代理链路保留。
- 管理 API 已增加统一 Bearer Token 认证，中间件会放行 `/health`，并对受保护接口的写操作和认证失败输出审计日志。
- 站点、证书、规则、代理参数和 SafeLine 运行时配置已支持刷新后即时生效；监听地址与监听器重建类参数仍按接口提示处理。
- 已补后端集成测试、管理 API 认证测试、管理 API 触发热更新测试，以及前端 Bearer Token 持久化测试。
- 管理台主要文案已从 “七层检测” 收口为 “HTTP 接入/代理” 或 “HTTP 规则”。

## 1. 目标与边界

根据当前确认，本项目后续定位调整为：

- 本项目负责网关接入、协议接收、站点路由、TLS/证书管理、基础 L4 防护、事件持久化、管理台与 SafeLine/雷池协同。
- 复杂 L7 检测能力不再由本项目承担，交由下游雷池 WAF 处理。
- 本项目后续重点投入在：
  - 管理面安全
  - 运行时热更新
  - 更真实的测试体系

明确不作为当前阶段重点的内容：

- 部署与运维文档
- 更细的观测与治理
- 安全边界说明文档

## 2. 总体改造方向

### 2.1 L7 职责收缩

保留：

- HTTP/1.1、HTTP/2、HTTP/3 请求接入与统一抽象
- 反向代理与回源逻辑
- Host/SNI 站点匹配
- 与下游雷池 WAF 的协同链路
- 必要的请求元数据补充，如 `x-forwarded-for`、`x-request-id`

删除或下沉：

- 当前 `L7Inspector` 中的检测语义
- 所有“本项目自己做七层攻击检测”的设计预期
- 管理台中容易让人误以为本项目具备完整 L7 检测能力的配置项和文案

调整原则：

- 保留协议解析，不保留本地七层检测引擎。
- 保留 L7 规则代理链路中真正必须的“路由前处理”，删除“安全检测器”定位。
- 所有 UI、API、配置字段都要反映新的职责边界，避免误导。

## 3. 接下来要做的事情

以下任务按优先级排序。

### P0. 收缩 L7 能力边界

目标：让代码和产品表达与新定位一致。

状态：已完成

要做的事情：

- 删除或废弃 `src/l7/mod.rs` 中当前 `L7Inspector` 的检测逻辑。
- 从引擎调用链中移除“L7 检测器”这一层，只保留：
  - 请求解析
  - 请求标准化
  - 站点匹配
  - 回源前头部补充
  - 规则链中确实需要保留的非检测型处理
- 评估并清理以下内容是否仍然需要：
  - `L7Config.http_inspection_enabled`
  - `max_request_size` 之外的“检测型”L7配置
  - L7 stats 页面中带有“检测能力”暗示的指标
  - 管理台 `AdminL7*` 页面与文案
- 如果仍保留 L7 页面，必须改名或改文案为：
  - 协议接入状态
  - 回源/健康检查状态
  - HTTP 协议栈配置
  - 而不是“七层检测”

建议交付结果：

- 删除或瘦身 `src/l7/mod.rs`
- 更新 `src/core/mod.rs`
- 更新 `src/core/engine.rs`
- 更新 `src/config/l7.rs`
- 更新 `src/api/settings_handlers.rs`
- 更新前端 `vue/src/pages/AdminL7Page.vue` 及相关组件文案

验收标准：

- 代码中不再把本项目表述为“L7 检测器”。
- 启动、代理、HTTP/1.1、HTTP/2、HTTP/3 链路不受影响。
- 管理台不再暗示本项目本地执行高级 L7 攻击检测。

### P0. 为管理面增加安全控制

目标：避免管理 API 成为高风险入口。

状态：已完成

要做的事情：

- 为管理 API 增加统一认证中间件。
- 首选实现一种简单但可靠的方案：
  - Bearer Token
  - 或 Basic Auth
- 将 `/health` 与必要只读探针接口是否匿名访问单独决策。
- 对写操作接口增加更严格保护：
  - 规则增删改
  - 证书增删改
  - 站点增删改
  - 插件安装/上传
  - SafeLine 同步触发
- 增加基础审计日志：
  - 谁调用了什么接口
  - 是否成功
  - 来源 IP
- 把认证配置加入持久化配置或环境变量覆盖体系。

建议交付结果：

- API Router 增加统一认证层
- 增加认证配置结构体
- 前端请求层支持附带认证信息
- 对敏感接口进行权限门禁

验收标准：

- 未认证访问管理接口时返回 401/403。
- 已认证用户可正常完成现有管理流程。
- 写操作有审计日志记录。

### P1. 实现运行时热更新

目标：站点、证书、配置变更不再依赖重启才能生效。

状态：已完成

要做的事情：

- 保留当前已有的“规则热刷新”机制。
- 新增以下运行时刷新能力：
  - 站点配置热刷新
  - 证书热刷新
  - 网关路由表热刷新
  - SafeLine 相关配置热刷新
- 对 L4/L7 配置做分类：
  - 可运行时生效的参数，支持热更新
  - 需要重建监听器的参数，做显式提示
- 把当前 API 返回中的“重启后生效”改为更细粒度提示：
  - 立即生效
  - 部分立即生效
  - 需要重建监听器
- 为 `WafContext` 增加可安全替换的运行时状态容器。
- 为 `GatewayRuntime` 增加刷新入口，而不是仅在启动时加载一次。

建议交付结果：

- `gateway_runtime` 可热替换
- 证书解析器可更新
- 站点 CRUD 后自动刷新内存态
- 设置更新后触发对应刷新逻辑

验收标准：

- 新增/修改站点后，无需重启即可影响路由匹配。
- 更新证书后，新连接可拿到新证书。
- 规则、站点、证书三类变更都具备清晰的运行时行为。

### P1. 建立更真实的测试体系

目标：从“单元测试能过”提升到“关键链路真的可用”。

状态：已完成

要做的事情：

- 增加后端集成测试：
  - HTTP/1.1 接入到回源链路
  - HTTP/2 接入到回源链路
  - 站点匹配链路
  - TLS/SNI 证书选择链路
  - 配置热更新链路
  - 管理 API 认证链路
- 增加管理面前端测试，至少覆盖：
  - 登录/认证态
  - 站点与证书管理主流程
  - 规则管理主流程
- 增加端到端测试：
  - 管理台发起配置变更
  - 后端热更新
  - 实际流量命中新配置
- 调整 CI：
  - 保留现有 `fmt/check/test`
  - 增加 integration test job
  - 增加带 feature 组合的测试

建议交付结果：

- 新建 `tests/` 集成测试目录
- 前端引入测试框架
- CI 增加集成测试阶段

验收标准：

- 核心链路不只靠单元测试证明。
- 关键变更不会轻易破坏代理、站点匹配、认证和热更新。

### P2. 清理配置与页面结构

目标：让项目表达更聚焦，降低维护成本。

状态：已完成

要做的事情：

- 清理与“本地 L7 安全检测”强绑定的字段、页面、接口命名。
- 统一后台文案：
  - “L7 检测”改为“HTTP 接入/代理”
  - “检测配置”改为“协议/回源配置”
- 清理无实际意义的统计项或占位字段。
- 处理当前 feature 下的编译 warning，尤其是 HTTP/3 相关未使用项。

验收标准：

- 配置结构更贴合真实职责。
- 管理台表达不再混淆“网关能力”和“WAF 检测能力”。
- `cargo check --all-features` 无明显无用代码警告。

## 4. 推荐实施顺序

建议按以下顺序推进：

1. L7 能力边界收缩
2. 管理面安全
3. 站点/证书/配置热更新
4. 集成测试与 E2E 测试
5. 配置和页面清理收尾

原因：

- 先收缩职责，避免后续在错误方向继续加功能。
- 再补管理面安全，降低改造期间风险。
- 热更新依赖清晰的运行时职责划分，应放在边界收缩之后。
- 测试体系要覆盖前面三类改动，否则后续维护成本仍然高。

## 5. 建议拆分为的具体开发任务

可以拆成以下 issue 或里程碑：

- Milestone A: 移除本地 L7 检测定位
  - 删除 `L7Inspector` 检测逻辑
  - 修改配置和 UI 文案
  - 保留协议接入与代理链路

- Milestone B: 管理 API 安全化
  - 增加认证
  - 增加审计日志
  - 前端适配认证流程

- Milestone C: 网关运行时热刷新
  - 站点热刷新
  - 证书热刷新
  - 配置分类热刷新

- Milestone D: 测试体系升级
  - 后端集成测试
  - 前端流程测试
  - CI 扩展

## 6. 当前阶段完成标志

本轮改造完成后，应满足以下状态：

- [x] 项目不再承担复杂 L7 攻击检测职责。
- [x] 项目成为更清晰的“安全网关 + 管理面 + L4 基础防护 + 下游 WAF 编排接入”系统。
- [x] 管理面默认支持统一认证接入与审计日志。
- [x] 站点、证书、规则与大部分 HTTP 代理参数更新后可即时影响运行时行为。
- [x] 关键链路已具备后端集成测试与前端基础测试覆盖。
- 管理面具备基础安全控制。
- 站点、证书、规则至少具备主要运行时刷新能力。
- 核心能力由集成测试而非仅单元测试支撑。

## 7. 暂不做事项

以下内容本轮明确不做：

- 部署与运维文档建设
- 更细颗粒度观测治理能力
- 安全边界说明文档化

后续若项目进入正式生产化阶段，再视需要补充。
