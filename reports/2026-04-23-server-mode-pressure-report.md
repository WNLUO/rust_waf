# 服务器模式压测报告

时间：2026-04-23
测试机：85.149.219.14

## 结论

- 当前实例在这台 2 核 4G 测试机上，`server_mode` 初始即为 `balanced`，压测全过程也保持在 `balanced`。
- 本次测试的稳定极限值为 `1800 rps`（目标值），实际可达约 `1562.18 rps`。
- 最高档位下，正常请求成功率仍为 `100%`，但调度延迟已经明显上升，说明机器已接近饱和。
- `runtime_pressure_level` 最终升到 `elevated`，`runtime_defense_depth` 维持 `balanced`。

## 测试环境

- CPU：2 核
- 内存：约 `3.89 GiB`
- 上游：本地 200 响应桩服务
- 负载模型：`10%` 正常流量 + `90%` 攻击流量
- 单档持续时间：`15s`

## 观察

- `server_mode` 始终为 `balanced`，说明这台机器在当前测试模型下尚未跨入更激进的保护切档。
- 到 `1500/1800` 档位时，客户端已经出现明显排队和超时，说明实际瓶颈主要在吞吐和响应排队，而不是策略误判。
- 如果你希望把 `server_mode` 也压到 `conservative / survival`，下一轮建议提高队列压力或改成更重的攻击型负载。

## 建议

- 保留当前“服务器模式优先”的实现，作为默认策略层。
- 增加一组更激进的队列/攻击压力测试，专门验证 `conservative` 和 `survival` 的切换条件。
- 把这次的 `1800 rps` 作为当前环境下的保守上限参考值，不要直接当成绝对理论峰值。
