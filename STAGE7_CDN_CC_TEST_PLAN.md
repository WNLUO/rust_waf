# 阶段 7：CDN CC 复测与中文报告测试方案

## 1. 是否可以开始阶段 7

结论：可以开始，但开测前必须先完成代码同步和环境确认。

| 检查项 | 当前判断 | 说明 |
|---|---|---|
| 阶段 1 CPU 压力感知 | 已完成 | 已把 CPU 压力纳入运行时压力模型，适合观察 CPU 是否更早触发保护 |
| 阶段 2 早期防御 | 已完成 | L4 之后、昂贵 L7 之前已有 early defense，可观察 early drop 是否降低 CPU |
| 阶段 3 L7 轻量模式 | 已完成 | rich/core/minimal 已具备，适合观察高级 CC 下 CPU 成本是否下降 |
| 阶段 4 CDN 双身份联动 | 已完成 | 已区分 CDN peer 和真实用户，并向 L7 输出 route/host 收紧信号 |
| 阶段 5 事件持久化降级 | 已完成 | 高压下事件会聚合/瘦身，适合观察 SQLite 队列是否稳定 |
| 阶段 6 AI 临时策略闭环 | 已完成 | hit/outcome/续期/撤销链路已补齐，适合观察智能化策略效果 |
| 远端测试环境 | 可继续使用 | 保留 `/root/rust_waf_test`，不得删除 |
| 开测前置条件 | 必须确认 | 当前本地工作区改动需要完整同步到远端测试目录后再测 |

判断：阶段 7 可以进入“准备并执行复测”，不需要再补一个新的功能阶段。

## 2. 本次复测目标

| 目标 | 说明 |
|---|---|
| 对比 CPU | 看优化后常规 CDN CC、高级 CDN CC 是否仍然先打满 CPU，以及峰值/平均值是否下降 |
| 对比内存 | 看 512 MB 约束下是否仍无 OOM，事件聚合是否降低内存和队列压力 |
| 对比拦截链路 | 看 L4、early defense、L7 CC、AI 临时策略分别贡献多少 |
| 对比后端保护 | 看后端成功率、失败率、平均代理延迟是否改善 |
| 验证 CDN 场景 | 重点验证不能误封 CDN peer，而是基于真实用户 IP、route、host 收紧 |
| 验证智能化 | 看 AI 临时策略是否生成、命中、评估、续期或撤销 |

## 3. 基线与对比口径

已有旧基线来自前一次报告：

| 场景 | 旧基线 CPU 最高 | 旧基线内存最高 | 旧结论 |
|---|---:|---:|---|
| 常规 CDN CC | 约 201.89% | 约 84.2 MB | CPU 先满，内存未爆 |
| 高级 CDN CC | 约 202.99% | 约 180 MB | CPU 先满，内存未爆 |

本阶段默认做法：

| 对比方式 | 说明 |
|---|---|
| 主对比 | 使用旧报告基线 vs 当前优化后版本 |
| 可选强对比 | 如果需要完全同环境 A/B，需要提供或恢复“优化前代码版本/镜像/commit”再跑一次旧版本 |
| 当前结论边界 | 如果只用旧报告做基线，结论属于“和历史压测结果对比”，不是同一时刻双版本 A/B |

## 5. 开测前准备

| 步骤 | 动作 | 通过标准 |
|---:|---|---|
| 1 | 确认远端目录 `/root/rust_waf_test` 存在 | 目录存在，历史脚本和结果保留 |
| 2 | 同步当前本地工作区代码到远端 | 远端运行的是阶段 1-6 后的当前代码 |
| 3 | 在远端构建 Rust WAF | 构建成功，无编译错误 |
| 4 | 确认 2 核/512 MB 限制 | `docker stats` 或 cgroup 配置能看到限制 |
| 5 | 启动模拟后端和 Rust WAF | WAF 能代理到后端，`/metrics` 可访问 |
| 6 | 清理本次临时进程 | 不删除历史目录，只停止冲突进程 |
| 7 | 创建本次结果目录 | 例如 `cdn-report-artifacts/2026-04-19-stage7-*` |

特别注意：

| 风险 | 处理 |
|---|---|
| 远端仍是旧代码 | 必须重新同步/构建，否则测不到阶段 1-6 的优化 |
| 只用 `git pull` | 不推荐；当前本地有未提交工作区改动，应该以当前工作区内容同步 |
| 历史结果被覆盖 | 所有输出必须写入新时间戳目录 |
| 测试过程中 CPU 满载 | 属于预期，但要确保 SSH 会话和采集脚本不被杀死 |

## 6. 测试场景

### 6.1 健康检查

| 项目 | 参数 |
|---|---|
| 持续时间 | 30 秒 |
| 并发 | 低并发 |
| 目标 | 确认 WAF、后端、指标、日志正常 |
| 成功标准 | 无崩溃，后端成功响应，`/metrics` 有数据 |

### 6.2 常规 CDN CC

| 项目 | 参数 |
|---|---|
| 持续时间 | 90 秒 |
| 并发 | 512 |
| 特征 | 固定或少量 route，真实用户 IP 分布较普通 |
| 重点观察 | CPU 峰值、平均 CPU、early drop、L7 block、后端成功率 |
| 预期 | CPU 压力仍可能很高，但应更早进入保护，后端失败率下降 |

### 6.3 高级 CDN CC

| 项目 | 参数 |
|---|---|
| 持续时间 | 120 秒 |
| 并发 | 512 |
| 特征 | 分散真实 IP、随机 UA、多 route、多 header，模拟更像绕过型 CC |
| 重点观察 | L7 tracking mode、route/host 阈值收紧、AI 临时策略、SQLite 队列 |
| 预期 | CPU 峰值和 L7 统计成本下降，summary 事件增加，后端被保护 |

### 6.4 长时间高级 CDN CC

| 项目 | 参数 |
|---|---|
| 持续时间 | 10-30 分钟 |
| 并发 | 先从 256 开始，稳定后再到 512 |
| 目标 | 验证长期稳定性、队列、内存、AI 策略续期/撤销 |
| 是否必跑 | 建议跑，但如果只想快速验证，可先跳过 |

## 7. 采集指标

| 类别 | 指标 |
|---|---|
| 系统资源 | CPU 峰值、CPU 平均值、内存峰值、是否 OOM |
| WAF 入口 | total packets、blocked packets、L4/L7 block |
| 运行时压力 | `runtime_pressure_level`、`runtime_defense_depth`、CPU score |
| early defense | `early_defense.action`、early drop 数量 |
| L7 CC | challenge、block、delay、tracking mode |
| CDN 身份 | trusted CDN forwarded、unresolved、spoofed forwarded header |
| SQLite | queue depth、dropped events、summary 聚合事件 |
| AI 策略 | active、effective、harmful、hits、observations、extensions、revoked |
| 后端保护 | proxy success、proxy failure、平均代理延迟 |
