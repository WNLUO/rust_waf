## 7. 单核承载能力估算

为了更贴近“有效攻击”的实际成本，本轮采用：

```text
有效攻击总量 ≈ blocked_l7 + proxied_requests
```

含义：

- 算入真正进入 WAF 处理链并产生拦截或代理结果的请求
- 不把纯连接失败、客户端超时、连接重置直接算作“有效攻击处理”

按此口径估算：

| 场景 | 有效攻击总量 | 持续时间 | 总体有效攻击 TPS | 单核折算 TPS（按平均 CPU） | 单核极限 TPS（按峰值 CPU） |
|---|---:|---:|---:|---:|---:|
| 常规 CDN CC | `204311 + 377 = 204688` | 90 秒 | 约 2274 TPS | 约 1899 TPS/核 | 约 1129 TPS/核 |
| 高级 CDN CC | `196123 + 2334 = 198457` | 120 秒 | 约 1654 TPS | 约 921 TPS/核 | 约 845 TPS/核 |

更适合直接引用的口径：

| 指标 | 常规 CDN CC | 高级 CDN CC |
|---|---:|---:|
| 单核平均承载能力 | 约 1900 次有效攻击/秒 | 约 920 次有效攻击/秒 |
| 单核接近打满时承载能力 | 约 1100 次有效攻击/秒 | 约 845 次有效攻击/秒 |

## 8. TPS 与并发换算

换算公式：

```text
并发 ≈ TPS × 平均响应时间(秒)
```

本轮压测延迟：

| 场景 | p50 | p95 |
|---|---:|---:|
| 常规 CDN CC | 36.31 ms | 62.37 ms |
| 高级 CDN CC | 83.62 ms | 217.92 ms |

按单核极限 TPS 估算：

| 场景 | 单核极限 TPS | 按 p50 估算并发 | 按 p95 估算并发 | 建议口径 |
|---|---:|---:|---:|---|
| 常规 CDN CC | 约 1129 TPS | 约 41 并发 | 约 70 并发 | 建议按 `40-70 并发/核` 理解 |
| 高级 CDN CC | 约 845 TPS | 约 71 并发 | 约 184 并发 | 建议按 `70-180 并发/核` 理解 |

按单核平均 TPS 估算：

| 场景 | 单核平均 TPS | 按 p50 估算并发 | 按 p95 估算并发 |
|---|---:|---:|---:|
| 常规 CDN CC | 约 1899 TPS | 约 69 并发 | 约 118 并发 |
| 高级 CDN CC | 约 921 TPS | 约 77 并发 | 约 201 并发 |

需要注意：

- `TPS` 和 `并发` 不是一个指标
- 高级场景并发区间更大，不代表更轻，而是因为请求耗时更长
- 如果讨论容量规划，建议始终同时给出 `TPS + 延迟 + 并发区间`

## 9. 关于“能否做到 2000+ TPS”的判断

### 9.1 结论

| 问题 | 判断 |
|---|---|
| Rust 语言本身能不能支持 2000+ TPS | 能 |
| 当前这套 WAF 在 2 核 512 MB 下能否稳定支撑高级 CDN CC `2000+` 有效攻击 TPS | 现在还不能 |
| 常规 CDN CC 是否已经接近或达到 2000+ TPS | 已达到 |
| 高级 CDN CC 继续优化后是否有机会逼近 2000+ TPS | 有机会 |
| 当前最大瓶颈是不是 Rust 语言本身 | 不是，主要是当前热路径过重 |

### 9.2 为什么不是 Rust 本身的问题

当前更像是“热路径成本问题”，而不是“语言上限问题”。真正决定上限的是：

| 热点因素 | 说明 |
|---|---|
| 频繁字符串处理 | route / host / UA / header / metadata 的构造与复制 |
| 多阶段决策链路 | 真实 IP 识别、L4、early defense、L7 CC、行为分析、AI、事件写入依次发生 |
| 频繁 Map / hash 开销 | 多维 bucket、计数、key 归一化、统计结构访问 |
| 高压模式仍不够便宜 | survival 模式已经变轻，但还不够“战时化” |
| 非核心观测混入热路径 | metadata、事件、AI 反馈、细粒度 reason 仍有成本 |

### 9.3 为什么说“常规场景已接近，复杂场景还差一截”

| 场景 | 当前有效攻击 TPS | 对 2000+ TPS 的判断 |
|---|---:|---|
| 常规 CDN CC | 约 2274 TPS | 已达到目标区间 |
| 高级 CDN CC | 约 1654 TPS | 还差约 20% 左右 |

所以：

- 常规攻击形态下，系统已经摸到 `2000+ TPS`
- 高级 CDN CC 下，仍需要再做一轮明确的热路径优化

## 10. 如果继续冲击 2000+ TPS，建议优先做什么

按 Rust 性能优化角度排序：

| 优先级 | 方向 | 目的 |
|---|---|---|
| P0 | 把高压热路径压成 `parse -> key -> counter -> verdict` | 降低每请求 CPU 成本 |
| P0 | 减少 `String`、`JSON`、`metadata` 动态构造 | 直接减少分配和拼接成本 |
| P0 | route / host / IP key 轻量化 | 降低 map 查找和 hash 成本 |
| P0 | 让 survival 模式更“战时化” | 进一步减少复杂 L7 和行为分析成本 |
| P1 | 把 AI 反馈、事件、统计从主热路径再解耦 | 降低尾延迟和抖动 |
| P1 | 对 repeated offender 做更激进缓存判定 | 降低重复计算成本 |
| P1 | 对高级 CC 的随机 UA / header 干扰做更强忽略策略 | 避免脏维度拖慢整体处理 |
| P2 | 基于 profile 定位热点函数，再做函数级细化优化 | 榨出最后的性能空间 |

## 11. 本轮过程中的实现性经验

这次执行阶段 7 时还沉淀了几个工程层面的注意事项：

| 项目 | 经验 |
|---|---|
| 远端构建方式 | 在服务器宿主机构建比容器内 512 MB 限制下构建更稳 |
| cargo 缓存 | 远端 `cargo-home` 已有有效缓存，挂载正确后编译速度可接受 |
| 二进制替换 | 宿主机构建产物要同步到容器实际挂载的 `target/release/waf` |
| data 目录 | 只用 Git 归档同步代码时，容易漏掉被忽略的 `data/` 运行时目录，需要额外保留 |
| 容器网络 | 复测容器要使用与历史环境一致的 host network 口径，否则宿主机上 `127.0.0.1` 无法直接访问 metrics / 压测入口 |
